{% extends "base.html" %} {% block title %}Demograder{% endblock %} {% block content %} Background The Demograder was written to provide a functionality that does not exist in most (all) autograders: the ability to run student code against other student code. Such a functionality may be useful in several ways, such as evaluating student code against each other's testcases as a way to develop good test-writing abilities, or to allow students to use each other's libraries to reinforce the importance of good API design and documentation. Supporting this core functionality of the Demograder drives the majority of its design decisions. Basic Usage Internally, the Demograder represents Course, Assignment, and Question objects. Courses represent a class that one or more instructors teach, which multiple students can enroll in. Assignments serve only a way to organize and group Questions, which represent the actual prompts that student code is responding to. Beyond due dates, timeout duration, grading script, and other common attributes shared with other autograders, Questions in the Demograder also have dependencies, which are other Questions whose submissions are used to evaluate this question. This dependency structure is used for both running student code against each other, as well as the more mundane task of running against instructor testcases. As an example, consider how we might set up the Demograder for a basic hello() function, that takes a string name and returns "Hello, " followed by that name. Two Questions are needed for this: one for hello.py that students would submit, and one for testcase.py that would import the student code and check that its output is correct for a specific input. Crucially, this second Question would be a dependency for the first one. Graphically, we might depict this relationship as: 

testcase.py
    ^
    |
    |
    |
 hello.py
What this means functionally is that, when a student submits to hello.py, the Demograder will also include the submissions from testcase.py in the same directory, which the instructor can then call. Concretely, a hello.py submission from the student might look like: 

# hello.py

def hello(name):
    return 'Hello, ' + name
A testcase.py submission from the instructor might look like: 

# testcase.py

from hello import hello

assert hello('Eve') == 'Hello, Eve'
And finally, the grading script for the hello.py Question might look like: 

#!/bin/sh

python3 testcase.py
As mentioned above, when this script is run to evaluate the student submission, both hello.py and testcase.py will be in the same directory, thus allowing Python to find and import them as needed. The Demograder only cares that the evaluation script returns an exit code of 0 (to indicate that the test passed), or any other exit code (to indicate the test failed); much more complicated scripts can be used as long as this basic API is respected. What if we want to add another testcase? To do so, the instructor would simply submit again to the testcase.py question. (The Demograder will rename the submitted files during evaluation, so the instructor can keep them named testcase1.py, testcase2.py, etc. locally.) By default, the Demograder will use all submissions from the dependencies and treat them as separate testcases. If a student submits hello.py now, it will be run against both testcase1.py and testcase2.py, resulting in a score out of 2. Advanced Usage: Supporting Library Files The basic idea behind how the Demograder works - that submissions to dependencies will be present during evaluation - easily supports other use cases. As an example, what if a third file `library.py` is also needed for evaluation? In that case, the instructor will create a third Question, and also set `hello.py` to depend on it (as well as `testcase.py`). Graphically, the dependencies could be depicted as: 

testcase.py      library.py
          ^      ^
           \    /
            \  /
             \/
           hello.py
By default, when a student submits `hello.py`, the combination of all submissions to both `testcase.py` and `library.py` will be used. As an example, if there were two submissions to `testcase.py` and three submissions to `library.py`, the student's `hello.py` submission would have a total of 2×3=6 results. Note that dependencies are not commutative. That is, if `library.py` itself has additional dependencies, the submission to those dependencies are not included in the directory when `hello.py` is evaluated. This allows for "circular" dependencies between student testcases and code, which can be used to evaluate each other. {% endblock %}
